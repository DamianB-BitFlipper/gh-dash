To use `dehub`, follow these steps after you've [installed it][01]:

1. Run

   ```bash
   gh dehub
   ```

2. Press <kbd>?</kbd> for help.
3. Run `gh dehub --help` for more info:

   ```bash
   Usage:
     gh dehub [flags]

   Flags:
     -c, --config string   use this configuration file (default lookup: a .dehub.yml file if inside a git repo, $DEHUB_CONFIG env var, or if not set, $XDG_CONFIG_HOME/dehub/config.yml)
         --debug           passing this flag will allow writing debug output to debug.log
     -h, --help            help for Dehub
   ```

## Flags

### `--config`

Specify the path to a configuration file to use for the dashboard. If the configuration file
doesn't exist or is invalid, `dehub` returns an error.

```bash
gh dehub --config path/to/configuration/file.yml
```

| Aliases |  Type  | Default                                                                                                                   |
| :------ | :----: | :------------------------------------------------------------------------------------------------------------------------ |
| `-c`    | String | `.dehub.yml` file if inside a git repo, `$DEHUB_CONFIG` env var, or if not set, `$XDG_CONFIG_HOME/dehub/config.yml` |

If you don't specify this flag, `dehub` uses the default configuration.
If the file doesn't exist, `dehub` will create it.

The location of the default configuration file depends on your system:

1. If `$DEHUB_CONFIG` is a non-empty string, `dehub` will use this file for
   its configuration.
2. If `$DEHUB_CONFIG` isn't set and you're in a git repository, it will look for `.dehub.yml` or `.dehub.yaml`
   in the repository root.
3. If neither of the above applies, then:
   - If `$XDG_CONFIG_HOME` is a non-empty string, the default path is `$XDG_CONFIG_HOME/dehub/config.yml`.
   - If `$XDG_CONFIG_HOME` isn't set, then:
     - On Linux and macOS systems, the default path is `$HOME/.config/dehub/config.yml`.
     - On Windows systems, the default path is `%USERPROFILE%\.config\dehub\config.yml`.

For more information about authoring configurations, see [Configuration][02].

### `--debug`

Specify whether `dehub` should write logs to the `debug.log` file in the current directory. By
default, `dehub` doesn't output debug information.

```bash
gh dehub --debug
```

| Aliases |  Type   | Default |
| :------ | :-----: | :------ |
| (None)  | Boolean | `false` |

When you use this flag, `dehub` creates the `debug.log` file in the current directory if it doesn't exist. If the file does exist, `dehub` appends new log entries to it.

### `--help`

Use this flag to display the help information for `dehub` in the terminal. If you specify this
flag, `dehub` ignores all other flags. It only displays the help information.

```bash
gh dehub --help
```

| Aliases |  Type   | Default |
| :------ | :-----: | :------ |
| `-h`    | Boolean | `false` |

### `--version`

Use this flag to display the version information for `dehub` in the terminal. If you specify this
flag with the `--config` or `--debug` flags, `dehub` ignores them. It only displays the version
information.

```bash
gh dehub --version
```

| Aliases |  Type   | Default |
| :------ | :-----: | :------ |
| `-v`    | Boolean | `false` |

When you use this flag, `dehub` emits the following information:

```text
dehub version <version>
commit: <commit_sha>
built at: <build_timestamp>
built by: <build_user>
goos: <operating_system>
goarch: <cpu_architecture>
```

- `<version>` is the extension's semantic version without a `v` prefix.
- `<commit_sha>` is the exact commit SHA the extension was built from.
- `<build_timestamp>` is the UTC date and time when the extension was built.
- `<build_user>` is who built the extension. For official releases, this is always `goreleaser`.

For example, the version information for the [v3.7.7 release][03] on Windows with an x64 processor
is:

```text
dehub version 3.7.7
commit: 6ce3f89ab0d73dd88e359133699d1cf920f88699
built at: 2023-04-15T08:42:16Z
built by: goreleaser
goos: windows
goarch: amd64
```

## Default Keybindings

When you use `dehub`, it displays the dashboard as a terminal UI (TUI). In the TUI, you can use
several commands by pressing key combinations to navigate and interact with the dashboard.

You can press the <kbd>?</kbd> key to toggle the help menu in the UI. The help menu lists the available
keybindings for the current context.

You can press the <kbd>q</kbd> key to quit the dashboard and return to your normal terminal view.

For more information about the keybindings for the dashboard, see [Keybindings][04].
